# Vibe Coding 的基本思路和步骤

## 概述

本篇随笔主要介绍了一种常见的 Vibe Coding 思路，包括需求确定、功能设计、任务划分和代码实现四大阶段，每个阶段则针对 AI 扮演的角色来设计提示词

## 1. 需求确定：确定自己要做什么

对于任何无法精准描述的需求，和 AI 讨论来确定需求文档。通常可以遵循以下四步：
- 目标：我要实现什么功能
- 输入：现在已有哪些文件/目录/其他资源
- 输出：需要 AI 在哪里输出什么样需求文档
- 步骤：以提问的方式帮助我确认需求，不要猜测意图，不明确的地方必须提问

同时，这四个步骤也可以作为通用的提示词结构，根据开发每个阶段的目标修改内容

## 2. 功能设计：讨论需求如何实现
#### （1）概要设计：划分模块，确定模块之间的联系
#### （2）详细设计：实现细节，完善每个模块的功能

进行设计的流程和上一步 **需求** 的确定类似，在具体设计时，应确保模块之间相互独立，可以分别进行单元测试

> 与 AI 对话时应尽量保持上下文简短，一方面减少 Token 消耗，另一方面能尽可能避免 AI 出现幻觉。追求每个会话只解决一个问题。


## 3. 任务划分：告诉 AI 如何写代码

AI 在根据需求和设计文档生成代码时通常将较多任务集中在同一个对话的上下文之中，这样不仅会拉长上下文，影响生成速度，消耗 Token，还会因为压缩上下文导致某些代码细节在后续的生成中被忽略，

基本思路如下：
- 目标：为每个模块划分最小可执行任务
- 输入：需求文档（path），详细设计文档（path）
- 输出：任务列表（path），每个模块对应一个 `<module-name>.md`，总体进度对应一个 `progress.md`
- 步骤：根据需求文档和详细设计，为每一个模块生成 Vibe Coding 所需要的最小任务。每个模块对应一个 `<module-name>.md`，用 check list 表示子任务是否完成；progress.md 中用 check list 表示模块是否已经完成

## 4. 代码实现：让 AI 完成开发

前文提到我们为了节省时间和成本，会尽量避免在同一个对话中完成所有代码，那么就需要将对话拆分成一个监工 Agent 和多个实现代码的子 Agent，由监工 Agent 为每个子 Agent分配要完成的具体最小可执行任务，最终实现完整功能的开发。同样的，我们可以让 AI 来生成这一复杂的提示词，用于指导和搭建这样的多层 Agent

生成提示词的提示词可以参考下面的思路：
- 目标：生成 Vibe Coding 所需要的提示词
- 输入：需求文档（path），详细设计文档（path），任务划分（path）
- 输出：提示词文档（path）
- 步骤：阅读输入信息，了解当前要实现的工程，生成提示词文档（path）作为 Vibe Coding 的起始提示词。主 Agent 用来跟踪项目整体的进度，生成子 Agent，用子 Agent 实现每一个模块，并完成测试。代码必须有完整的单元测试，并通过其他语法检查（根据语言而定）。生成提示词的过程中，如果有任何不不明确的地方都必须再次向我提问

在这个提示词生成的过程中，我们加入了最后一次提问讨论的机会，确保最终结果符合我们的预期，同时要求 AI 生成的代码必须经过多重测试，一定程度减少了 AI 幻觉等导致的代码不规范。通常 AI 会生成较长的复杂详细的提示词文档，但我们仍需要审核这一文档，以确保 AI 会按照我们的目标执行任务。
